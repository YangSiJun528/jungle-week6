# Codex 프롬프트 작성 피드백 요약

## 목적

- 이 문서는 비교 결과를 다시 길게 설명하는 대신, 앞으로 프롬프트를 더 잘 쓰기 위한 피드백만 추려서 정리한 문서다.

## 팀원별 총평과 피드백

### sijun

- 총평: 요구사항을 가장 세밀하게 통제하는 스타일이다. 그래서 테스트, 구조, 문법 규칙이 코드에 강하게 반영되는 장점이 있다.
- 피드백: 가장 좋은 점은 `수정 대상 + 제약 + 기대 동작`을 같이 주는 것이다. 이 패턴은 계속 유지하는 것이 좋다.
- 피드백: 구현 요청과 `commit/push/PR` 요청을 같은 흐름에 자주 섞는데, 이 둘을 분리하면 Codex가 더 안정적으로 한 작업에 집중할 수 있다.
- 피드백: 같은 요구를 다시 반복하기보다 "직전 결과에서 무엇이 부족했고 이번에는 무엇만 바꾸면 되는지"만 추가하는 방식이 더 효율적이다.

### wei

- 총평: 구현 전에 비교와 합의를 먼저 만드는 스타일이다. 네 명 중 가장 설계 지향적이고, 프롬프트와 코드 변화의 연결도 가장 깔끔한 편이다.
- 피드백: 좋은 점은 선택지를 먼저 비교하고 팀 설명용 근거까지 확보하는 것이다. 합의가 필요한 작업에서 특히 강한 방식이다.
- 피드백: 구현이 시작된 뒤에는 짧은 확인 프롬프트를 여러 번 나누기보다, 허용 문법과 성공 기준을 한 번에 묶어 주면 더 좋다.
- 피드백: 설명 문서 요청과 기능 구현 요청을 분리하면 산출물 우선순위가 더 선명해지고, 구현 흐름도 덜 끊긴다.

### seonho

- 총평: 실제 실행 결과를 보고 바로 다시 요청하는 반응형 스타일이다. 사용자 관점의 문제를 빨리 집어내는 장점이 있지만, 요구가 흔들리기 쉬운 편이다.
- 피드백: `컴팩트하게`, `짧게`, `main 안에` 같은 표현은 방향은 주지만 성공 기준이 약하다. 대신 `파일 수`, `허용 함수 수`, `테스트 방식`, `입출력 형식`처럼 구체적 제약으로 바꾸는 것이 좋다.
- 피드백: 실행 오류를 줄 때는 지금처럼 로그를 주는 것이 좋다. 여기에 `기대했던 출력`까지 같이 적으면 수정 방향이 훨씬 빨리 맞춰진다.
- 피드백: 환경 문제, 실행 문제, 기능 요구, push 요청을 한 흐름에 섞지 말고 분리해야 한다. 지금은 이 혼합 때문에 구현 목표가 자주 흔들린다.

### haegeon

- 총평: 계획과 문서를 먼저 만들고 그 다음 구현하는 스타일이 가장 뚜렷하다. 큰 작업을 안전하게 진행하기 좋은 방식이다.
- 피드백: `AGENTS.md`, `GOAL.md`, 계획 문서를 먼저 기준으로 삼는 습관은 매우 좋다. 이런 방식은 복잡한 작업에서 특히 효과적이다.
- 피드백: 다만 중간에 `main.c 하나로 단순화`와 `별도 파일로 다시 분리`가 반복되면 Codex 입장에서는 방향이 자주 바뀐 것으로 해석된다. 구조 원칙은 초반에 더 빨리 고정하는 편이 좋다.
- 피드백: 구현 문제와 git, 인증, 발열, Codex 설정 같은 주변 이슈를 분리하면 실제 코드 작업의 연속성이 더 좋아진다.

## 전체에게 해당되는 피드백 3가지

### 1. 한 번의 프롬프트에 최소 3가지는 항상 넣기

- 무엇을 바꿀지
- 어디를 바꿀지
- 어떻게 되면 성공인지

- 예시: `wei/main.c의 parse를 수정해서 SELECT * from {table}; 만 허용하게 하고, 세미콜론이 없으면 PARSE_ERROR를 반환하게 해줘. 테스트도 같이 맞춰줘.`

### 2. 구현 요청과 운영 요청을 섞지 않기

- `구현`
- `리뷰`
- `문서화`
- `commit/push/PR`

- 이 단계들을 한 프롬프트에 다 넣으면 Codex가 지금 무엇을 우선해야 하는지 흐려진다.
- 가장 좋은 방식은 `구현 -> 확인 -> git 작업` 순서로 나누는 것이다.

### 3. 안 된다고 말할 때는 재현 정보까지 같이 주기

- 입력 예시
- 실행 명령어
- 실제 오류 로그
- 기대했던 결과

- 이 네 가지가 같이 있으면 가장 빠르게 수정된다.
- 반대로 `안 돼`, `에러가 남`, `다시 해줘`만 있으면 Codex는 다시 추측을 많이 해야 한다.

## 한 줄 결론

- 좋은 프롬프트는 길어서 좋은 것이 아니라, `수정 대상`, `제약`, `성공 기준`, `재현 정보`가 같이 들어 있어서 바로 코드로 옮길 수 있는 프롬프트다.
