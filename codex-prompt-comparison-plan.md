# Codex 프롬프트 사용 방식 비교 기준

## Summary

- 자료는 사용자가 직접 작성한 프롬프트 기록과 그 직후의 git 커밋, 코드 변화만 사용한다.
- 목적은 평가가 아니라 누가 어떤 방식으로 Codex를 썼고 그 방식이 어떤 결과를 만들었는지 비교하는 것이다.
- 점수나 등급 대신 비교 축과 대표 사례 중심으로 정리한다.
- 비교 대상은 `sijun`, `wei`, `seonho`, `haegeon` 네 명 전체다.

## Source Map

- `sijun`: `sijun/prompt/sijun-log.jsonl`
- `wei`: `wei/logs/user-prompts.jsonl`
- `seonho`: `seonho/log.md`
- `haegeon`: `haegeon/analysis/prompt-log/prompt_events_full.jsonl`
- 각 팀원은 자기 폴더 관련 git 커밋과 현재 코드 구조를 함께 본다.

## Inclusion Rules

- 사용자 프롬프트만 비교 대상으로 삼는다.
- 자동 생성 제목 프롬프트, 훅 테스트, 단순 잡담/노이즈, 계획 Markdown 작성 요청은 제외한다.
- `haegeon` 로그는 `prompt_origin=user_prompt`만 사용하고, `analysis_scope=exclude` 행은 제외한다.
- 짧은 후속 프롬프트라도 실제 구현/수정 흐름에 영향을 주면 에피소드 맥락 안에서만 참고한다.
- 결과 비교는 반드시 네 명 모두를 포함해야 하며, 특정 팀원을 보류 상태로 남기지 않는다.

## Comparison Axes

### 1. 프롬프트 스타일 비교

- 처음부터 요구사항을 구조화해서 쓰는지
- 짧게 던지고 여러 번 수정하는지
- 코드 작성 요청보다 설명이나 질문 비중이 큰지
- 구현 제약을 많이 거는지
- 에러가 났을 때 로그와 재현 정보를 같이 주는지

### 2. 프롬프트의 구체성 비교

- 목표가 분명한지
- 수정 대상이 분명한지
- 성공 기준이 있는지
- 테스트나 실행 방법이 포함되는지
- `간단하게`, `컴팩트하게` 같은 추상 표현이 많은지, 아니면 동작 기준이 명확한지

### 3. 프롬프트-코드 연결 방식 비교

- 프롬프트 하나가 바로 커밋으로 이어지는지
- 여러 번의 추가 프롬프트를 거쳐서 구현되는지
- 프롬프트에서 말한 제약이 실제 코드 구조에 반영되는지
- 설명 요청이 많고 실제 구현 연결은 약한지
- 환경 문제 해결 프롬프트가 구현 흐름을 자주 끊는지

### 4. 결과물 성향 비교

- 단일 파일로 뭉치는 경향이 있는지
- 모듈 분리를 선호하는지
- 테스트를 빨리 붙이는지
- REPL, 입력 처리, 파일 저장 같은 요구가 코드에서 얼마나 직접적으로 반영되는지
- 수정이 누적되면서 구조가 안정되는지, 아니면 계속 임시 대응이 붙는지

## Good vs Weak Prompt Patterns

### 좋은 프롬프트의 특징

- 무엇을 만들지와 무엇은 하지 말아야 하는지가 같이 있다.
- 파일, 함수, 입출력, 테스트 기준이 구체적이다.
- 이전 결과의 문제를 정확히 짚고 수정 방향을 준다.
- 실행 결과나 오류 메시지를 포함해 후속 요청을 한다.
- 최종 코드 형태를 상상할 수 있을 정도로 제약이 선명하다.

### 아쉬운 프롬프트의 특징

- 요구는 많은데 성공 기준이 없다.
- 추상적 표현이 많고 실제 수정 범위는 불명확하다.
- 구현 문제와 환경 문제를 한 흐름에 계속 섞는다.
- `안 된다`는 피드백은 있는데 재현 정보가 부족하다.
- 앞선 요청과 충돌하는 제약이 뒤늦게 추가된다.

## Comparison Process

- 먼저 팀원별 로그 파일 위치와 형식을 확인한다.
- 프롬프트를 시간순으로 정리한다.
- 자동 생성 제목 프롬프트, 훅 테스트, 계획 Markdown, 노이즈성 입력은 제외한다.
- 비슷한 주제의 프롬프트를 하나의 작업 에피소드로 묶는다.
- 각 에피소드마다 다음을 정리한다.
  - 처음 요청이 어땠는지
  - 중간 수정 요청이 어떻게 바뀌었는지
  - 어떤 커밋이나 코드 변화로 이어졌는지
  - 그 과정에서 드러난 사용 특징이 무엇인지
- 마지막에 네 명 전체를 같은 형식으로 맞춰 주요 스타일, 강한 패턴, 반복되는 약한 패턴을 비교한다.

## Output Format

팀원별로 아래 형식으로 맞춘다.

- 프롬프트 사용 스타일 한 줄 요약
- 대표적인 좋은 프롬프트 패턴
- 대표적인 아쉬운 프롬프트 패턴
- 프롬프트가 코드로 이어지는 방식
- 전체 비교에서 보이는 특징

## Assumptions

- 비교 대상은 Codex 사용 방식이지, 최종 코드 실력 자체만은 아니다.
- 같은 기능을 만들었더라도 프롬프트 방식이 다르면 그 차이를 중심으로 본다.
- 로그 형식이 팀원마다 달라도, 비교 기준과 제외 규칙은 동일하게 유지한다.
