# Codex 프롬프트 사용 방식 비교 결과

## 분석 범위와 적용 규칙

- 기준 문서는 `codex-prompt-comparison-plan.md`를 그대로 따랐다.
- 프롬프트 로그는 `sijun/prompt/sijun-log.jsonl`, `wei/logs/user-prompts.jsonl`, `seonho/log.md`, `haegeon/analysis/prompt-log/prompt_events_full.jsonl`를 사용했다.
- 코드 변화는 각 폴더 기준 git 커밋과 현재 파일 구조를 함께 봤다.
- 자동 생성 제목 프롬프트, 훅 테스트, 단순 노이즈, 계획 Markdown만 저장해 달라는 요청은 비교 본문에서 제외했다.
- `haegeon`은 `prompt_origin=user_prompt`만 사용했고, `analysis_scope=exclude`는 제외했다.
- 점수화는 하지 않았고, 누가 어떤 방식으로 Codex를 썼고 그 방식이 어떤 코드 결과로 이어졌는지를 비교 축 중심으로 정리했다.

## sijun

- 프롬프트 사용 스타일 한 줄 요약: 제약을 매우 촘촘하게 거는 편이고, 구현 후에도 테스트 구조, 추상화 레벨, 주석 형식, 문법 규칙까지 계속 세부 조정하는 방식이다.
- 대표적인 좋은 프롬프트 패턴: `getline()` 사용, `.exit` 메타 명령 처리, 단위/통합 테스트 분리, EBNF 명세 반영, `metadata.csv` 기반 로딩처럼 수정 대상과 성공 기준이 매우 구체적이다.
- 대표적인 좋은 프롬프트 패턴: 파싱 실패 시 stderr 출력, quoted string 강제, 세미콜론 강제처럼 실제 버그 증상과 기대 동작을 같이 적어 후속 수정 방향이 선명하다.
- 대표적인 아쉬운 프롬프트 패턴: `Implement the plan.`, `커밋 ㄱㄱ`, `push`류 요청이 자주 끼어들어 구현 맥락과 git 작업 맥락이 계속 섞인다.
- 대표적인 아쉬운 프롬프트 패턴: 같은 내용을 거의 같은 문장으로 다시 던지거나, `.gitignore`, 브랜치 정리, PR 정리 같은 운영 이슈가 구현 흐름을 자주 끊는다.
- 프롬프트가 코드로 이어지는 방식: 초기 CLI/테스트 논의와 AGENTS 제약은 `f4b947d`로 이어져 `main.c`, `README.md`, 테스트 파일이 한 번에 생겼다.
- 프롬프트가 코드로 이어지는 방식: 테스트 책임 분리 요구는 `5565299`로 이어져 `test_main_unit.c`, `test_main_integration.c`, `main.h`, `CMakePresets.json`이 추가되며 구조가 빠르게 테스트 중심으로 재편됐다.
- 프롬프트가 코드로 이어지는 방식: EBNF 기반 파서, 메타데이터 로더, 추상화 레벨별 분리는 `ebcaa93`, `41324a6`, `631349f`, `134e54f`로 이어져 `parser_level2.c`, `parser_level3.c`, `parser_level4.c`, `metadata.c`, `repl.c`까지 모듈이 크게 늘었다.
- 프롬프트가 코드로 이어지는 방식: CSV 실행과 출력 형식 요구는 `43d55f7`, `3c448b1`, `baec43a`, `142df49`, `ed718b4`로 이어져 `executor.c`, `db/*.csv`, SQL 문서, 프롬프트 표시, quoted string 규칙, metadata 기반 초기화까지 붙었다.
- 전체 비교에서 보이는 특징: 네 명 중 프롬프트 제약이 가장 강하다. 무엇을 만들지뿐 아니라 어떤 파일에 둘지, 어떤 수준으로 나눌지, 어떤 테스트가 책임질지까지 직접 통제한다.
- 전체 비교에서 보이는 특징: 그 결과 현재 코드는 가장 빠르게 단순 스캐폴드에서 다모듈 구조로 커졌고, 테스트 비중도 가장 강하게 반영됐다.
- 전체 비교에서 보이는 특징: 반면 구현 요청과 git 작업 요청이 자주 뒤섞여 흐름이 길어지고, 실제 작업 에피소드 하나가 여러 운영 프롬프트에 의해 잘게 끊기는 경향이 있다.

## wei

- 프롬프트 사용 스타일 한 줄 요약: 구현 전에 선택지를 비교하고 합의안을 만든 뒤, 확정된 계획을 기준으로 단계별로 실행하는 설계 지향형 사용 방식이다.
- 대표적인 좋은 프롬프트 패턴: `getopt`, 표준입력 리다이렉션, `assert+셸 스크립트`를 비교해 달라고 하거나, 팀원 설명용 한 문장 주장과 근거를 요구하는 식으로 먼저 판단 기준을 세운다.
- 대표적인 좋은 프롬프트 패턴: `wei` 폴더만 수정, 파일 두 개만 생성, 세미콜론 규칙 강제, `SELECT * from {table_name};`만 허용처럼 수정 범위와 허용 문법을 구체적으로 제한한다.
- 대표적인 좋은 프롬프트 패턴: 동료 코드 리뷰 내용과 실제 스모크 테스트 결과를 다시 넣어 quoted CSV 처리, 대소문자/공백 허용, 표 형태 출력까지 후속 개선으로 연결한다.
- 대표적인 아쉬운 프롬프트 패턴: 구현 중간에 주석 문서화, 원리 설명용 Markdown, push/PR 확인 요청이 자주 들어와 본 기능 개발과 학습용 산출물이 계속 섞인다.
- 대표적인 아쉬운 프롬프트 패턴: 짧은 확인 프롬프트가 많아 한 에피소드가 잘게 쪼개진다. 다만 대부분 직전 코드의 특정 줄이나 특정 규칙을 가리켜 맥락 손실은 상대적으로 적다.
- 프롬프트가 코드로 이어지는 방식: 초기 main 단계 계획 확정과 `wei` 폴더 두 파일 제약은 `6a9cd2e`로 이어져 `wei/main.c`, `wei/test`의 최소 REPL 뼈대가 만들어졌다.
- 프롬프트가 코드로 이어지는 방식: parse 단계에서 허용 문법을 다시 좁히고 주석/설명 문서를 요구한 흐름은 `1b4bd5d`, `1a755cf`로 이어져 `main.c` 안에 SQL 파서가 커지고 대소문자와 공백 처리 규칙이 완화됐다.
- 프롬프트가 코드로 이어지는 방식: 실행/저장 단계는 "핵심 로직은 새 파일"이라는 지시를 그대로 따라 `e7a9c2d`에서 `execute.c`가 생겼고, 이어 `2771e5b`에서 따옴표 안 쉼표를 안전하게 처리하는 보정이 추가됐다.
- 전체 비교에서 보이는 특징: 네 명 중 계획 수립과 기준 합의에 가장 많은 프롬프트를 쓴다. 구현 전에 비교, 합의, 의사코드, 설명 문서를 먼저 만드는 비중이 높다.
- 전체 비교에서 보이는 특징: 코드 결과도 그 성향을 반영해 초기에는 최소 파일로 시작하지만, 실행 단계에서만 별도 `execute.c`를 도입하는 식으로 확장 포인트를 선택적으로 분리한다.
- 전체 비교에서 보이는 특징: 프롬프트와 코드의 연결이 비교적 깔끔하다. 동료 피드백이나 테스트 로그가 들어오면 바로 그 규칙이 다음 커밋의 핵심 수정으로 반영된다.

## seonho

- 프롬프트 사용 스타일 한 줄 요약: 큰 명세를 먼저 던지지만 실제 구현 단계에서는 "컴팩트하게", "main 안에", "짧게" 같은 압축 요구와 실행 오류 피드백이 반복되는 반응형 사용 방식이다.
- 대표적인 좋은 프롬프트 패턴: `select`가 실제 데이터를 보여야 한다, CSV 파일이 있어야 한다, assert와 리다이렉트 방식이어야 한다처럼 사용자 입장에서 보이는 결과를 직접 지적하는 프롬프트는 방향을 분명히 만든다.
- 대표적인 좋은 프롬프트 패턴: 실행 로그, `ENOENT` 오류, 실제 터미널 명령, 스크린샷 경로를 함께 준 구간은 문제 재현에 필요한 정보가 포함되어 있다.
- 대표적인 아쉬운 프롬프트 패턴: 환경 문제, 실행 오류, 구현 요구, 구조 선호, push 요청이 짧은 프롬프트로 연속 섞여 나와 한 에피소드의 목표가 자주 흔들린다.
- 대표적인 아쉬운 프롬프트 패턴: `컴팩트하게`, `짧게`, `main 함수 안에`, `다 합쳐봐` 같은 추상 제약은 강하지만 정확한 성공 기준은 뒤늦게 붙는 경우가 많다.
- 프롬프트가 코드로 이어지는 방식: 사이클 명세와 compact 구현 요구는 `4308311`, `92424e5`, `ae55af1`로 이어져 `seonho/cycle_compact/main.c`, `test.c`, `input.sql` 중심의 작은 구조가 먼저 잡혔다.
- 프롬프트가 코드로 이어지는 방식: parse error와 assert/리다이렉트 요구가 이어진 뒤 `7d1598f`에서 compact REPL parser가 추가됐고, 이후 한국어 설명 보강이 `fadf72c`로 반영됐다.
- 프롬프트가 코드로 이어지는 방식: "select가 실제 저장값을 보여야 한다", "csv 같은 게 있어야 한다"는 요구는 `8a2f6ae`로 이어져 단순 CSV 실행 흐름이 붙었다.
- 프롬프트가 코드로 이어지는 방식: 반복된 실행/구조 설명 요구는 마지막에 `3171cd7`의 `README.md`, `log.md`로 정리됐다.
- 전체 비교에서 보이는 특징: 네 명 중 가장 단일 파일 지향이 강하다. 현재 결과물도 `cycle_compact/main.c` 중심이고, `test.c`와 샘플 데이터가 같은 폴더에 붙는 구조다.
- 전체 비교에서 보이는 특징: 결과물이 빠르게 눈에 보이는 데모 형태로 수렴하지만, 그만큼 구조 선택이 설명 부족이나 환경 이슈에 크게 흔들린다.
- 전체 비교에서 보이는 특징: 프롬프트 하나가 곧바로 안정된 커밋으로 가기보다, 실행 오류와 재요구를 여러 번 거쳐 겨우 다음 단계로 넘어가는 경우가 많다.

## haegeon

- 프롬프트 사용 스타일 한 줄 요약: 문서와 계획을 먼저 만들고 승인한 뒤 구현하는 방식이 가장 뚜렷하며, AGENTS/GOAL 같은 기준 문서를 자주 직접 참조한다.
- 대표적인 좋은 프롬프트 패턴: `AGENTS.md`와 `GOAL.md`를 읽고 계획부터 세우라고 하거나, "구현하지 말고 plan만 다듬어라"처럼 실행 조건을 명확히 나누는 프롬프트가 많다.
- 대표적인 좋은 프롬프트 패턴: parse 리팩터링 피드백, 실패 예시 문서 오류 지적, `select` 실행 오류 로그처럼 코드 리뷰나 문서 검토 결과를 다시 구체적으로 전달한다.
- 대표적인 아쉬운 프롬프트 패턴: 구현 철학이 중간에 크게 흔들린다. 처음에는 `main.c` 하나, `.h` 삭제, 단순 MVP를 원하다가 나중에는 리뷰 편의를 위해 별도 파일 분리를 다시 요구한다.
- 대표적인 아쉬운 프롬프트 패턴: git push, PR, 인증, 발열, Codex 설정 같은 주변 이슈 질문이 자주 끼어들어 본 구현 흐름을 여러 번 멈춘다.
- 프롬프트가 코드로 이어지는 방식: 첫 단계는 `main`에서 입력만 받는 MVP 요구가 `21d833b`, `ec80944`로 이어져 아주 작은 입력 루프와 종료 조건이 먼저 만들어졌다.
- 프롬프트가 코드로 이어지는 방식: parse 단계는 계획 문서 생성 후 승인하는 흐름을 거쳐 `001a72d`, `00f7e64`, `23ecf5d`로 이어졌고, 결과적으로 `parse.c`, `parse.h`, `test_parse.c`가 큰 비중을 차지하게 됐다.
- 프롬프트가 코드로 이어지는 방식: execute 단계도 먼저 계획 문서(`08a6852`)를 만든 뒤 `1d6963d`에서 구현됐고, 이후 "main에 함수가 너무 많다"는 리뷰성 프롬프트가 `03f0ef4`의 `cli.c`, `execute.c`, `execute.h` 분리로 이어졌다.
- 전체 비교에서 보이는 특징: 네 명 중 문서 선행 비중이 가장 높다. 현재도 `docs/plan_2_parser.md`, `docs/plan_3_execute.md`, `docs/test-strategy-notes.md` 등 문서량이 매우 많다.
- 전체 비교에서 보이는 특징: 결과물은 가장 전통적인 모듈 분리 구조에 가깝다. `cli`, `parse`, `execute`가 나뉘고 테스트도 parse/execute 별로 크게 분리되어 있다.
- 전체 비교에서 보이는 특징: 다만 프롬프트 차원에서는 단순화 욕구와 구조화 욕구가 번갈아 나와, 실제 구현이 한 번 main에 몰렸다가 다시 분리되는 왕복이 발생했다.

## 전체 비교에서 보이는 공통점과 차이

- `sijun`은 제약을 가장 세게 건다. 파일 위치, 테스트 책임, 문법 규칙, 추상화 깊이까지 프롬프트에서 직접 설계한다.
- `wei`는 구현 전에 비교와 합의부터 한다. 그래서 프롬프트가 가장 "설계 문서 -> 계획 승인 -> 구현" 흐름에 가깝다.
- `seonho`는 사용 중 드러난 문제를 빠르게 다시 던지는 편이다. 그래서 프롬프트는 가장 반응형이고, 실행 오류와 구조 재요구가 많이 섞인다.
- `haegeon`은 문서 승인 절차가 가장 강하다. 계획 Markdown과 기준 문서 참조가 실제 구현 단계의 전제처럼 쓰인다.
- 결과물 관점에서 `seonho`는 가장 단일 파일 지향이고, `sijun`과 `haegeon`은 가장 모듈 분리와 테스트 분리가 강하다. `wei`는 그 중간에서 핵심 분리만 선택적으로 적용했다.
- 좋은 프롬프트 패턴은 네 명 모두에서 "수정 대상이 분명하고, 실패 증상이나 기대 동작이 같이 적힌 경우"였다. 이런 프롬프트는 거의 항상 다음 커밋의 핵심 변경으로 바로 연결됐다.
- 약한 프롬프트 패턴은 "구현과 운영 작업을 섞는 경우", "성공 기준 없는 압축 표현만 남는 경우", "계획 저장이나 push 요청이 구현 흐름을 자르는 경우"였다.
- 종합하면, Codex를 잘 쓴 흔적은 길고 화려한 프롬프트보다도 `무엇을 바꾸는지`, `어디를 바꾸는지`, `어떻게 확인할지`가 동시에 들어간 프롬프트에서 가장 선명하게 나타났다.
