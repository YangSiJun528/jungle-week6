# 수요코딩회 발표 보고서

## 1. 시연 + 구현 소개


> 작성 예정
>
> 실제 발표 직전에 데모 순서와 구현 설명 흐름에 맞춰 보완 예정

## 2. 협업 방식

이번 수요코딩회에서는 `AI로 빠르게 만든 결과물`보다 `AI를 사람이 얼마나 통제할 수 있는지`를 중심에 두고 진행했음.

기능 자체를 뽐내기보다, AI를 활용하는 방식과 그 과정을 비교해보는 실험에 가까웠음.

- 같은 주제를 4명이 각자 구현
- 구현 결과를 서로 코드 리뷰
- 프롬프트 로그를 모아서 비교
- 어떤 방식이 더 통제 가능했는지 회고

## 3. 기존의 문제점 인식

### 3-1. AI 결과물이 금방 블랙박스가 됨
- 결과는 나오지만, 왜 이렇게 구현됐는지 설명이 어려움
- 코드가 조금만 길어져도 사람이 전체 흐름을 놓치기 쉬움
- 결국 "돌아가긴 하는데 이해는 잘 안 되는 코드"가 남기 쉬움

### 3-2. 결과물이 비슷해짐
- 큰 요구사항을 그대로 던지면 비슷한 프롬프트, 비슷한 코드, 비슷한 발표가 나옴
- 팀별 차이가 줄어들고, 서로에게서 배울 수 있는 포인트도 약해짐

## 4. 목표

이 문제 인식을 바탕으로 아래 목표를 설정했음.

> AI를 통제하기 위해, 프롬프트를 더 잘 쓰고 사람이 이해 가능한 범위 안에서 개발

## 5. 도전 시도

### 5-1. 짧은 코드로 각자 구현하고 코드 리뷰하기

이번에 가장 크게 시도한 실험은 `짧은 코드 + 각자 구현 + 코드 리뷰`였음.

**선택 이유**
- AI를 어떻게 쓰는지에 따라 결과물이 얼마나 달라지는지 확인 필요
- 코드가 짧다면 서로 읽고 리뷰하는 것도 가능하다는 판단
- 각자 구현하면 프롬프트 스타일과 설계 선택 차이가 더 잘 드러날 것이라는 기대

### 5-2. 프롬프트 로그 수집하고 분석하기

작성한 프롬프트를 모두 남기고, 이후 비교와 피드백에 활용하는 방식이었음.

**선택 이유**
- 개선을 위해서는 측정 가능한 근거가 필요했음
- 어떤 요청이 좋았는지, 어떤 요청이 비효율적이었는지 기록이 필요했음
- 감으로 회고하지 않고 실제 기록을 바탕으로 이야기하고 싶었음

그래서 작업 과정에서 나온 프롬프트를 모아 아래 작업을 진행했음.

- 서로 비교
- AI에게 다시 보여주고 피드백 확보

정리 결과는 [Codex 프롬프트 작성 피드백 요약](./codex-prompt-feedback-summary.md)에 반영했음.

## 6. 진행 방식

개념 공부 이후, 작은 범위로 구현과 확인을 반복하는 방식으로 진행했음.

### 6-1. 개념 공부 + 큰 틀 구상

개념과 기술적 선택에 대한 공통 질문을 각자 준비하고 공유하면서 학습했음.

이후 전체적인 구현 계획을 세우고, 어떤 단위로 사이클을 돌릴지 먼저 정했음.

### 6-2. 공부 -> 개발 -> 리뷰 사이클 반복

구현 단계에서는 `공부 -> 개발 -> 리뷰` 사이클을 반복했음.

| 단계 | 한 일 |
| --- | --- |
| 공부 | 핵심 개념과 최소 명세 정리 |
| 개발 | 각자 작은 범위로 구현하면서 AI 사용 |
| 리뷰 | 코드와 프롬프트 로그를 비교하고 피드백 |

한 번에 완성도를 높이기보다, 작은 단위로 구현하고 바로 확인과 리뷰를 붙이는 흐름이 핵심이었음.

## 7. 결과 정리

### 7-1. 좋았던 점

- 새로운 협업 방식을 실제로 시도

- GitHub PR 리뷰 방식으로 협업 경험 축적

### 7-2. 아쉬웠던 점

- 생각보다 전체 소요 시간이 길었음
- GitHub을 통한 협업 방식에 적응하는 데 예상보다 시간이 더 들었음
- 코드 리뷰에 드는 비용이 예상보다 컸음
- 코드 레벨까지 충분히 이해하지 못한 상태에서 리뷰하게 되는 경우가 있었음
- 후반부로 갈수록 `코드를 짧게 쓰자`, `AI를 잘 써보자`, `리뷰를 해보자`보다 `일단 빨리 끝내자`가 더 커졌음
- 그러다 보니 생각보다 코드를 짧게 유지하기도 어려웠음

## 8. 개선점 후보

다음에 다시 한다면 아래 방향이 더 현실적이라고 판단했음.

### 8-1. 병렬 구현보다 페어 프로그래밍 활용
- 명세는 같이 작성
- 구현은 한 명이 대표로 담당
- 나머지는 페어 혹은 리뷰 중심으로 참여

### 8-2. 시작 전에 최소 버전부터 뽑아보기

- AI에게 `최소 기능만 남긴 가장 짧은 버전`을 먼저 요청
- 실제 코드 볼륨을 확인한 뒤 구현 범위 결정

### 8-3. 코드는 더 짧게, 리뷰 범위는 더 좁게

- 코드가 길어지는 것을 가장 큰 리스크로 인식
- 코어 기능만 남기고 나머지는 뒤로 미룸
- 리뷰도 전체 취약점 탐색보다 핵심 설계, 흐름, 명세 준수 여부 위주로 진행

### 8-4. 프롬프트 로그 수집은 유지하기

- 비용이 거의 없음
- 회고 자료로 바로 활용 가능
- AI에게 다시 피드백받는 용도로도 유용함

이 부분은 이번 방식에서 비교적 부담이 적고 효과가 좋았던 요소였음.

## 9. 결론

이번 수요코딩회는 `AI로 빨리 만드는 법`보다 `AI를 사람이 통제하면서 쓰는 법`을 실험한 시간이었음.

결과적으로 아래와 같은 결론.

- 시도 자체는 충분히 의미 있었음
- 다만 `각자 구현 + 전체 리뷰`는 비용이 너무 컸음
- 다음에는 구현은 더 집중시키고, 리뷰는 더 가볍고 선명하게 가져가는 편이 좋음
