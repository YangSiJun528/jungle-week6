## 협업 소개

이번 주 수요코딩회에서는 4명이 같은 주제를 각자 구현하면서, 단순히 AI로 결과물을 빨리 만드는 것이 아니라 `AI를 어떻게 써야 사람이 주도권을 가질 수 있는가`를 같이 실험해봤다.

핵심은 기능 구현 자체보다도,

- AI를 통제하려는 시도가 실제로 가능한지
- 그 시도가 협업과 리뷰에 어떤 영향을 주는지
- 무엇이 효과적이었고 무엇이 비효율적이었는지

를 직접 해보고 정리해보는 데 있었다.

## 기존의 문제점 인식

기존에는 AI에게 큰 요구사항을 던지고 빠르게 결과물을 받는 방식이 많았다. 이 방식은 속도는 빠르지만, 몇 가지 문제가 분명했다.

- 코드가 금방 블랙박스가 된다.
- 결과물은 나오지만 왜 이렇게 구현됐는지 설명하기 어렵다.
- 팀마다 비슷한 프롬프트, 비슷한 결과물, 비슷한 발표가 반복되기 쉽다.
- 코드가 커질수록 사람의 이해와 리뷰가 따라가지 못한다.

결국 `AI를 잘 쓴다`는 것이 단순히 결과물을 빠르게 뽑는 것과 같은가에 대한 의문이 생겼고, 이번에는 그 부분을 직접 검증해보기로 했다.

## 목표

이번 시도의 목표는 한 문장으로 정리하면 아래와 같다.

> AI를 통제하기 위해, 프롬프트를 더 잘 쓰고 사람이 이해 가능한 범위 안에서 개발한다.

조금 더 구체적으로는 다음을 목표로 했다.

- 사람이 먼저 구조와 방향을 정한다.
- AI는 그 방향 안에서 구현을 돕는 도구로 사용한다.
- 결과물뿐 아니라 프롬프트와 판단 과정도 같이 남긴다.
- 나중에 비교와 회고가 가능하도록 협업 흔적을 남긴다.

## 도전 시도

### 1. 코드량을 짧게 가져가서 각자 구현하고 코드 리뷰하기

같은 주제를 각자 따로 구현한 뒤, 서로의 결과물을 코드 리뷰하는 방식을 시도했다.

선택 이유는 명확했다.

- AI를 어떻게 쓰는지와 그 결과물이 실제로 어떻게 달라지는지 비교하고 싶었다.
- 코드량이 짧다면 서로의 코드를 읽고 리뷰하는 것도 가능하다고 판단했다.
- 각자 구현하면 프롬프트 스타일과 설계 선택의 차이가 더 잘 드러날 것이라고 봤다.

즉, `짧은 코드 + 별도 구현 + 코드 리뷰`를 통해 AI 사용 방식의 차이와 결과물의 차이를 함께 보려는 시도였다.

### 2. 프롬프트 로그를 수집하고 분석하기

개선하려면 측정할 수 있는 무언가가 필요하다고 판단해서, 프롬프트 로그도 따로 모았다.

선택 이유는 다음과 같았다.

- 무엇을 어떻게 요청했는지 남아야 회고가 가능하다.
- 좋은 결과가 나온 프롬프트와 아쉬운 프롬프트를 비교할 수 있다.
- 감으로만 회고하지 않고, 실제 작업 기록을 바탕으로 피드백할 수 있다.

그래서 프롬프트를 기록하고,

- 직접 서로 비교해보거나
- AI에게 로그를 보여주고 피드백을 받거나

하는 방식으로 회고 자료를 만들었다.

## 진행 계획

이번 진행은 `공부 -> 개발 -> 리뷰` 사이클을 돌리는 방식으로 잡았다.

1. 공부
   핵심 개념과 최소 명세를 먼저 맞추고, 무엇을 이해해야 하는지 정리한다.
2. 개발
   각자 작은 범위로 구현하면서 AI를 사용한다.
3. 리뷰
   코드와 프롬프트 로그를 바탕으로 비교하고 피드백한다.

이 사이클을 통해 단순 구현이 아니라, `이해`, `실험`, `회고`가 같이 남는 흐름을 만들고 싶었다.

## 결론

이번 시도는 완성도 높은 협업 방식이라고 보기는 어려웠지만, 분명히 의미는 있었다.

- AI를 통제하려는 시도를 실제로 해봤다.
- 프롬프트와 결과물의 관계를 비교해볼 수 있었다.
- 협업과 리뷰에서 무엇이 어려운지 구체적으로 드러났다.

반대로, 우리가 기대했던 `짧은 코드라서 리뷰가 쉬울 것이다`라는 가정은 현실에서 많이 깨졌다. AI를 더 잘 쓰기 위해서는 프롬프트만이 아니라 범위 설정, 리뷰 방식, 협업 구조까지 같이 설계해야 한다는 점이 드러났다.

## 장점

- 새로운 협업 방식을 실제로 시도해봤다.
- AI를 다루기 위한 교훈을 얻었다.
- 좋은 프롬프트는 길이가 아니라 `수정 대상`, `제약`, `성공 기준`이 중요하다는 점을 확인했다.
- 프롬프트 로그를 남겨서 회고 근거를 확보했다.
- GitHub PR 리뷰 방식으로 협업해본 경험이 남았다.

## 단점

- 생각보다 시간이 너무 오래 걸렸다.
- 초반에는 코드 레벨까지 충분히 이해하지 못한 채 리뷰해야 해서 인지 부하가 컸다.
- 리뷰에 드는 에너지를 과소평가했다.
- 각자 따로 개발하고 다시 서로 리뷰하는 구조 자체가 부담이 컸다.
- 초반 세팅과 GitHub 사용 방식에 익숙해지는 데도 시간이 들었다.

## 후기

좋았던 점은, AI의 문제를 막연하게 느끼는 데서 끝나지 않고 실제로 문제를 인식하고 개선을 시도했다는 것이다. 특히 프롬프트 로그를 남겨두니, 작업 내역과 연결해서 어떤 요청이 좋았고 어떤 요청이 비효율적이었는지 더 선명하게 볼 수 있었다.

아쉬웠던 점은 비용이 예상보다 훨씬 컸다는 것이다. 따로 개발하는 것도 부담이었고, 남의 코드를 읽는 것도 생각보다 훨씬 어려웠다. 특히 초반에는 코드가 복잡하게 느껴져 리뷰 자체가 큰 작업이 됐다.

## 개선점 후보

- 명세는 함께 쓰되, 구현은 한 명이 대표로 맡고 나머지는 페어 혹은 리뷰 중심으로 붙는 방식
- 드라이버와 내비게이터를 번갈아 맡는 페어 프로그래밍 방식으로, `구현 담당`을 매 사이클마다 순환하는 방식
- 처음 시작할 때 AI에게 `최소 기능만 남긴 가장 짧은 버전`을 먼저 뽑아 보고, 실제 코드 볼륨을 확인한 뒤 범위를 정하는 방식
- 코드 길어지는 것을 가장 큰 리스크로 보고, 코어 기능만 남기고 나머지는 과감히 뒤로 미루는 방식
- 코드 전체의 취약점 찾기보다 핵심 설계, 흐름, 명세 준수 여부 위주로 리뷰 범위를 줄이는 방식
- 프롬프트 로그 수집과 AI 피드백은 비용이 낮고 효과가 있었으므로 다음에도 유지하는 방식
- 각자 따로 작업하는 시간을 길게 가져가기보다, 같이 있는 시간에 프롬프팅과 리뷰를 더 많이 공유하는 방식

지금 기준으로 보면, `각자 따로 개발하고 서로 전부 리뷰하는 방식`은 학습 효과는 있었지만 투입 자원이 너무 컸다. 다음에는 비교 실험의 장점은 살리되, 구현은 더 집중시키고 리뷰는 더 가볍고 선명하게 만드는 방향이 필요하다.
