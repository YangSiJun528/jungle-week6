## 3. 협업 소개

이번 수요코딩회에서는 `같이 AI를 써서 빨리 만든다`보다, `AI를 사람이 주도하면서 써볼 수 있을까`를 실험해봤다.

정리하면 이런 협업이었다.

- 같은 주제를 4명이 각자 구현
- 구현 결과를 서로 코드 리뷰
- 프롬프트 로그를 모아서 비교
- 어떤 방식이 더 통제 가능했는지 회고

즉, 이번 발표의 포인트는 기능 자랑보다 `AI를 다루는 방식 자체를 실험한 기록`에 가깝다.

## 4. 기존의 문제점 인식

기존 방식에서 느낀 문제는 아래처럼 정리할 수 있었다.

### 4-1. AI 결과물이 금방 블랙박스가 된다

- 결과는 나오는데, 왜 이렇게 구현됐는지 설명하기 어렵다
- 코드가 조금만 길어져도 사람이 전체 흐름을 놓치기 쉽다
- 결국 "돌아가긴 하는데 이해는 잘 안 되는 코드"가 남기 쉽다

### 4-2. 결과물이 비슷해진다

- 큰 요구사항을 그대로 던지면 비슷한 프롬프트, 비슷한 코드, 비슷한 발표가 나온다
- 팀별 차이가 줄어들고, 서로에게서 배울 수 있는 포인트도 약해진다

### 4-3. 리뷰 비용을 쉽게 본다

- 코드가 짧을 줄 알고 시작했지만 실제로는 그렇지 않았다
- 내 코드도 봐야 하고 남의 코드도 봐야 해서 부담이 빠르게 커졌다
- 특히 초반에는 코드 자체를 이해하는 데 드는 인지 부하가 컸다

## 5. 목표

이번 시도의 목표는 아래 한 줄로 정리했다.

> AI를 통제하기 위해, 프롬프트를 더 잘 쓰고 사람이 이해 가능한 범위 안에서 개발한다.

세부 목표는 다음과 같았다.

- 사람이 먼저 구조와 방향을 정한다
- AI는 그 방향 안에서 구현을 돕는 도구로 쓴다
- 결과물뿐 아니라 프롬프트와 판단 과정도 남긴다
- 나중에 비교와 회고가 가능하도록 협업 흔적을 남긴다

## 6. 도전 시도

### 6-1. 짧은 코드로 각자 구현하고 코드 리뷰하기

이번에 가장 크게 해본 실험은 `짧은 코드 + 각자 구현 + 코드 리뷰`였다.

선택 이유는 이랬다.

- AI를 어떻게 쓰는지에 따라 결과물이 얼마나 달라지는지 보고 싶었다
- 코드가 짧다면 서로 읽고 리뷰하는 것도 가능하다고 봤다
- 각자 구현하면 프롬프트 스타일과 설계 선택 차이가 더 잘 드러날 것 같았다

기대한 효과는 아래와 같았다.

- AI 사용 방식 비교
- 결과물 비교
- 리뷰를 통한 학습
- GitHub 협업 경험 쌓기

### 6-2. 프롬프트 로그 수집하고 분석하기

또 하나의 실험은 `프롬프트 로그 수집`이었다.

이걸 넣은 이유는 단순했다.

- 개선하려면 측정할 수 있는 근거가 필요했다
- 어떤 요청이 좋았는지, 어떤 요청이 비효율적이었는지 남겨야 했다
- 감으로 회고하지 말고 실제 기록을 보고 이야기하고 싶었다

그래서 작업 과정에서 나온 프롬프트를 모아서,

- 서로 비교해보기도 하고
- AI에게 다시 보여주고 피드백을 받기도 했다

## 7. 진행 방식

전체 흐름은 `공부 -> 개발 -> 리뷰` 사이클로 진행했다.

| 단계 | 한 일 |
| --- | --- |
| 공부 | 핵심 개념과 최소 명세 정리 |
| 개발 | 각자 작은 범위로 구현하면서 AI 사용 |
| 리뷰 | 코드와 프롬프트 로그를 비교하고 피드백 |

원래 의도는 단순 구현이 아니라 `이해`, `실험`, `회고`가 같이 남는 흐름을 만드는 것이었다.

## 8. 결과 정리

### 8-1. 좋았던 점

- 새로운 협업 방식을 실제로 시도해봤다
- AI를 통제하려는 문제의식을 구체적인 방식으로 옮겨봤다
- 좋은 프롬프트는 길이보다 `수정 대상`, `제약`, `성공 기준`이 중요하다는 점을 확인했다
- 프롬프트 로그를 남겨서 회고의 근거를 확보했다
- GitHub PR 리뷰 방식으로 협업해본 경험이 남았다

> TODO: PR 리뷰 이미지 캡처 삽입

### 8-2. 아쉬웠던 점

- 생각보다 시간이 너무 오래 걸렸다
- 코드 레벨까지 충분히 이해하지 못한 상태에서 리뷰를 해야 했다
- 리뷰에 드는 에너지를 과소평가했다
- 각자 따로 개발하고 다시 서로 리뷰하는 구조가 생각보다 무거웠다
- 초반 세팅과 GitHub 사용 방식에 익숙해지는 시간도 꽤 들었다

### 8-3. 특히 크게 드러난 한계

- `코드가 짧으면 리뷰도 쉽다`는 가정이 실제로는 잘 맞지 않았다
- 병렬 구현 시간이 길어질수록 서로의 맥락이 벌어졌다
- 후반부로 갈수록 `AI를 잘 써보자`보다 `일단 빨리 끝내자`가 더 커졌다

## 9. 후기

이번 시도는 완성된 방식이라기보다, `어디서 막히는지 직접 확인한 실험`에 가까웠다.

좋았던 쪽은 분명했다.

- AI의 문제를 막연하게 느끼는 데서 끝나지 않았다
- 직접 기록하고 비교하면서 개선 포인트를 찾을 수 있었다
- 프롬프트 로그를 보고 작업 내역과 연결해서 회고할 수 있었다

아쉬운 쪽도 분명했다.

- 따로 개발하는 부담이 컸다
- 남의 코드를 읽는 데 예상보다 훨씬 많은 에너지가 들었다
- 코드가 조금만 복잡해져도 리뷰 자체가 큰 일이 됐다

## 10. 개선점 후보

다음에 다시 한다면, 아래 방향이 더 현실적이라는 결론이 나왔다.

### 10-1. 병렬 구현보다 대표 구현 중심으로 가기

- 명세는 같이 쓴다
- 구현은 한 명이 대표로 맡는다
- 나머지는 페어 혹은 리뷰 중심으로 붙는다

이 방식이 더 효율적인 이유는,

- 구현 분산은 줄이고
- 이해는 같이 가져가고
- 리뷰 포인트는 더 선명하게 만들 수 있기 때문이다

### 10-2. 페어 프로그래밍을 돌려가면서 해보기

- 드라이버와 내비게이터를 번갈아 맡는다
- 한 사람만 계속 구현하지 않도록 사이클마다 역할을 바꾼다
- 프롬프팅 경험도 돌아가면서 가져간다

이렇게 하면 각자 따로 멀어지는 시간보다, 같이 맥락을 공유하는 시간이 늘어난다.

### 10-3. 시작 전에 최소 버전부터 뽑아보기

- 처음부터 끝까지 다 만들려고 들어가지 않는다
- AI에게 `최소 기능만 남긴 가장 짧은 버전`을 먼저 뽑아본다
- 실제 코드 볼륨을 확인한 뒤, 이번 범위를 정한다

이 단계가 먼저 있었으면,

- 리뷰 가능한지
- 구현량이 감당 가능한지
- 어디까지가 핵심인지

를 더 빨리 판단할 수 있었을 것 같다.

### 10-4. 코드는 더 짧게, 리뷰 범위는 더 좁게

- 코드 길어지는 것을 가장 큰 리스크로 본다
- 코어 기능만 남기고 나머지는 뒤로 미룬다
- 리뷰도 전체 취약점 찾기보다 핵심 설계, 흐름, 명세 준수 여부 위주로 본다

### 10-5. 프롬프트 로그 수집은 유지하기

- 비용이 거의 없다
- 회고 자료로 바로 쓸 수 있다
- AI에게 다시 피드백받는 용도로도 유용하다

이 부분은 이번 방식에서 비교적 부담이 적고 효과가 좋았던 요소였다.

## 11. 결론

이번 수요코딩회는 `AI로 빨리 만드는 법`보다 `AI를 사람이 통제하면서 쓰는 법`을 실험한 시간이었다.

결과적으로,

- 시도 자체는 충분히 의미 있었다
- 다만 `각자 구현 + 전체 리뷰`는 비용이 너무 컸다
- 다음에는 구현은 더 집중시키고, 리뷰는 더 가볍고 선명하게 가져가는 편이 좋다
